# 랜덤포레스트가 뭐야?

의사결정트리를 여러 개 만들어서
여러 트리들의 투표로 결과를 결정하는 알고리즘이야

# 트리를 여러 개 만든다고? 왜?

하나의 의사결정트리는 사실 좀 멍청해
데이터가 조금만 바뀌어도 결과가 확 바뀌는 과적합 문제가 심해

그러면 트리가 특정 데이터에만 잘 맞는 모델이 돼버려서 실제로는 성능이 떨어져

그래서 "그냥 하나 말고 여러 개 만들면 어때?"라는 아이디어가 나온 거야
랜덤포레스트 = 여러 트리의 평균 / 투표 → 훨씬 안정적

# 트리를 많이 만든다고 똑똑해지나?

그냥 많이 만든다고 되는 건 아니고
트리마다 서로 다른 관점을 갖게 만들어야 해

그래야 한 트리가 실수해도 전체가 실수할 확률이 줄어들거든

# 그럼 서로 다르게 만든다는 게 뭔 말이야?

두 가지 랜덤성을 넣어:

1. Bootstrap(부트스트랩) – 데이터를 랜덤하게 샘플링

트리마다 원본 데이터 대신 "뽑은 데이터"로 학습해

예:

원본 데이터: 10만 건

트리1: 랜덤하게 10만 건에서 뽑은 샘플

트리2: 또 랜덤하게 다른 샘플

→ 트리들이 서로 다른 데이터 관점을 갖게 돼

2 Feature Randomness(특징 랜덤 선택)

트리가 분기(split)할 때 모든 컬럼을 보지 않아

예:

모든 컬럼: 50개

split 시 랜덤하게 7개만 보고 최적 분기 결정

→ 트리들 사이 공통점이 줄어듦
→ 트리들이 다양해지고, 그만큼 “평균”이 강력해져

# 그럼 예측은 어떻게 해?

분류(Classification): 여러 트리가 투표해서 가장 많은 표가 나온 라벨 선택

회귀(Regression): 각 트리의 예측을 평균냄

즉,
“한 명의 전문가” 말보다
“여러 전문가의 다수 의견”이 더 정확하다는 논리야

너가 면접 보러 갔다고 생각해봐

1명의 면접관: 편향도 있고 기분 따라서도 흔들리고 정확도도 낮아

20명의 면접관: 한 명이 이상한 판단을 해도 전체 결과는 안정적

랜덤포레스트는 “면접관 100명을 모아놓고 다수결 내리는 구조”라고 보면 돼

# 단점도 있어?

당연히 있어

개별 트리가 너무 깊으면 전체도 무거워지고 느려짐

모든 트리가 독립적으로 예측하니 실시간 요청에 비효율적일 수 있음

“왜 이런 결과가 나왔는지” 해석이 어려움(블랙박스)

하지만!

데이터 전처리가 크게 필요 없고

범주형,수치형 다 잘 처리하고

기본값만 써도 성능이 매우 준수

그래서 실무에서 진짜 많이 쓰는 모델

# 문자열 같은 건 어떻게 다루는데?

랜덤포레스트 자체는 문자열을 직접 비교하지 않아
트리 알고리즘은 숫자 비교 기반이거든

그래서 문자열은 보통

Label Encoding (문자열 → 숫자)

One-hot encoding (카테고리 → 벡터)

이런 식으로 사전 변환해서 넣어

여기서도 너가 말한 것처럼 내부적으로는 결국 숫자 비교야

한 줄로 요약하면?

“랜덤하게 만든 여러 개의 의사결정트리를 모아서
다수결로 똑똑한 예측을 한다”
라는 구조야
